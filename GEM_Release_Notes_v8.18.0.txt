1. MAJOR PRODUCT CHANGES IN v8.18.0
--------------------------------------------------------------------------------

Decision trust & status
  * Canonical FIVE-state Final Decision Status on every hand (gem_final_status):
        MISTAKE > CONDITIONAL > CLEARED > UNASSESSED ("Not reviewed") >
        UNGRADED ("No decision").
    Derived once and consumed everywhere — the lazy hand body, the static shell
    and the hand-list popup all read the SAME status (0 status contradictions
    over 844 hands). A gradeable hand with no adjudication is UNASSESSED, never
    CLEARED; a result-only hand is UNGRADED and carries no strategic grade.
  * REV17 action-sizing & ownership truth preserved end to end: every rendered
    sized action equals the canonical full-history replay (3177 sized actions,
    0 raw-sizing fallbacks; frozen Stage-F gates intact).

Commentary
  * Commentary Capsule with VISIBLE registers — Fact / Coach / Insufficient
    evidence (no silent "Unclear"); an insufficient-evidence capsule states the
    actual limitation.
  * Full-population zero-drop ledger: every source commentary item lands on a
    named visible surface (2721 items, balances, 0 silent drops).

Tournament Results  (one canonical typed DataTable, one Results nav destination)
  * Finish presentation: every event with a valid finish + entries shows
    place / field + Top%. "No cash" is a RETURN outcome (Return column) and
    NEVER replaces the finish.
  * Average Top% is computed over EVERY event with a valid Top% (not only
    cashes) = 48.1% on the June-16 session (12 events).
  * BB/100 and cEV/100 render as non-currency signed numbers (-8.9 / +97.3),
    never as currency.
  * Sticky speed filters — Regular / Turbo / Hyper — with recomputing totals.
  * Exit hand is the final column, rendered via PokerHandDisplay; Top% to one
    decimal; -100% ROI muted but retained.

Hands & villains
  * PokerHandDisplay (gem_report_draft/_cards): one typed card/hand component
    reused across the header, the hand-list popup and the Results exit hand;
    the hand id is kept separate from the cards; same component desktop + mobile.
  * Villain Teaching: explicit SEVEN-part lessons (what the villain did / cue /
    read / confidence / current exploit / future exploit / guardrail) with a
    stable identity (tournament_id|player_hash). ELIGIBILITY is decided
    INDEPENDENTLY of completeness: a lesson missing only its future exploit is
    eligible-but-incomplete, never reclassified ineligible.

Packaging
  * Lean Claude-Chat runtime (report generation only) — ~30% smaller than the
    full source bundle. The verification/test/acceptance apparatus (acceptance/,
    _qa_*, _test_scratch.py) ships only in the FULL release bundle.

2. REQUIRED WORKFLOW (Claude Chat project)
--------------------------------------------------------------------------------

  STEP 0 (once per session — rebuild the runtime from the lean package):
      python /mnt/project/gem_lean_runtime.py /home/claude/gem
      cd /home/claude/gem
      cp /mnt/project/session_*.csv .
      pip install phevaluator        # exact EAI / True-EV
      python -c "import gem_report_draft, gem_analyzer; print('runtime OK')"

  Generate a report from a hand-history folder:
      python gem_analyzer.py <SESSION_DIR> <SESSION_NAME> --player <NAME> --quick
      # full analyst pass (also emits the analyst worklist):
      python gem_analyzer.py <SESSION_DIR> <SESSION_NAME> --player <NAME>

  Output: ONE static, portable HTML report (desktop + mobile read the same hand
  data). No feature recalculates sizing / status / tournament truth / hand markup
  independently.

--------------------------------------------------------------------------------
4. KNOWN ISSUE — Villain Teaching future-exploit wording
   (ACCEPTED BY THE PRODUCT OWNER AS NON-BLOCKING FOR v8.18.0)
--------------------------------------------------------------------------------

  Some generated Villain Teaching future-exploit ("Next time …") sentences may:
    - be truncated by the future-exploit word cap; and/or
    - use an archetype-level adjustment that is less closely aligned to the
      specific observed cue.

  Scope / impact: WORDING QUALITY ONLY. Lesson eligibility, completeness counts,
  current-exploit guidance, villain chronology and no-hindsight protections are
  NOT affected.

  PLANNED POST-RELEASE FIX (BACKLOG ITEM #1 — the FIRST bounded bug-fix after
  v8.18.0): raise/lift the future-exploit word cap so sentences are not
  truncated, and tighten cue-specific alignment (prefer a cue-derived future
  adjustment over an archetype-level template), with a regression test that
  every eligible future-exploit is complete (untruncated) and cue-aligned.
