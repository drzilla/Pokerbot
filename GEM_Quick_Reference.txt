**STEP 0 — REBUILD THE PACKAGE FIRST (DO THIS BEFORE ANYTHING ELSE)**

Before reading further in this file, and before running any pipeline
step, read `SESSION_START_STEP0_package_rebuild.txt` and follow it: the
`gem_report_draft` renderer ships as **16** loose files and must be
reassembled into a `gem_report_draft/` package. The 16th file is
`sections_issue_explorer.py` (v8.1.0). Verify the rebuild —
`import gem_report_draft` and `import gem_analyzer` must both print OK —
before continuing with the rest of this Quick Reference.

---

**DEFAULT ACTION — HAND-HISTORY FILE ATTACHED WITH NO INSTRUCTION**

A hand-history file attached on its own, with no other text, IS the
instruction: run the full GEM pipeline end-to-end — `gem_run.py`
(parse → analyze → render HTML + MD) — then the analyst pass per
`Analyst_Writing_Checklist.md`, then return the finished report and its
output file locations. No need to ask what to do.

---

**ANALYST WRITING CHECKLIST — READ FIRST**

Before writing any session_analysis_*.json entry, read
`Analyst_Writing_Checklist.md`. Catches the recurring mistakes:
TPTK vs TPGK, flush-count miscounts, range definitions missing or
prose-blob format, equity computed vs starting range instead of
narrowed range, blockers stated as buckets instead of specific
combos, "Ron asked" reclassification attribution.

Hard rule: ≤300 words per argument. No tournament name. No chip
amounts. No restated grid info.

FLAG REVIEW PROTOCOL (v7.20 — MANDATORY for every session)
========================================================

The analyzer produces CANDIDATES, not mistakes. Candidates become
confirmed mistakes only after review.

For each candidate in the "Needs Review — Claude Scrutiny Required"
section of Del 4B, Claude applies this framework BEFORE marking
anything as a mistake:

1. VILLAIN'S RANGE at the decision point
   - What's villain's effective stack? (NOT their chip stack — effective.)
   - What's the jam context tag? (standard / squeeze / cover / micro)
   - What's their range GIVEN the context?
   - If the context widens the range vs the Quick Ref chart, USE the wider
     range, don't apply the base chart.

2. POT ODDS + EQUITY vs the widened range
   - What equity does Hero need to call/raise profitably?
   - What's Hero's actual equity vs the correctly-sized range?
   - If bounty format + Hero covers: add 3-10% EV premium (see bounty
     cover cheat sheet above).

3. MULTI-STREET PLAN with decision-tree branches
   - If action is preflop-only (all-in or all-in-response), state the
     equity math and conclude.
   - If action has postflop: write the plan for each street with key
     runout branches (brick turn vs scare turn, paired river, etc.).

4. CLASSIFICATION → one of:
   - CONFIRMED MISTAKE: flag stands. Real leak.
   - OK / CORRECTLY PLAYED: play is +EV given correct range/context.
     Auto-corrected by reviewer or verified by Claude scrutiny.
   - VARIANCE LOSS: play was +EV, ran into bad runout. NOT a leak.
   - DETECTOR BUG: the analyzer flagged something it shouldn't have.
     Auto-logged to pipeline change requests.

5. If RE-CLASSIFIED as OK or Detector Bug:
   - Write a pattern signature entry to gem_exceptions_log.csv
   - Future sessions will auto-match and skip re-diagnosing

IMPORTANT: Del 1F (Top 3 Leaks), Del 7 (Action Card), and the GTO
shortlist draw ONLY from the CONFIRMED bucket. Never from raw
detector output.

When scrutinizing hands conversationally (not just in the report),
Claude uses the SAME format: (1) villain range, (2) multi-street plan
with decision tree. This applies to ALL hand discussions, not just
formal drill-mode.

SHOVE VERDICT PREFLIGHT (v7.47 — MANDATORY for every all-in)
========================================================

Before classifying ANY shove/all-in decision — Hero shoves, Hero
called all-in, Hero faces an all-in — Claude MUST emit both
preflight blocks below BEFORE any range analysis or verdict. This
applies to live conversational hand review AND report generation,
not just one or the other.

The hard rule "use effective stack" and "apply bounty cover premium"
existed prior to v7.47 but allowed implicit reasoning. The KTo CO
15BB miss (Amit hand review, 2026-05-12) traced to skipping the
explicit write-down. These preflights force the structured output.

--------------------------------------------------------
PREFLIGHT-1: EFFECTIVE STACK DECLARATION
--------------------------------------------------------

Required format (one line, before any range analysis):

    [Eff Stack] Hero=X.XX, opponents in pot + yet-to-act:
                {pos: stack, pos: stack, ...},
                EFF = min(...) = Y.YY BB

Rules:
  • "Yet-to-act" means players who haven't folded AND haven't acted
    yet at the decision moment. Folded players are excluded. Players
    behind who could still enter the pot are INCLUDED — they
    constrain the effective stack of the decision even if they
    end up folding.
  • For a shove decision, EFF determines max Hero can win per caller.
    A Hero stack > shortest yet-to-act stack means Hero COVERS that
    opponent — relevant for bounty matrix (PREFLIGHT-2).
  • Never skip the write-down. The point is the forcing function;
    Claude has already shown the rule alone isn't enough.

Worked example (KTo CO 15BB hand, 2026-05-12):
  [Eff Stack] Hero=15.01, in pot: UTG=49.68 HJ=57.31,
              yet-to-act: BTN=13.92 SB=56.69 BB=29.17,
              EFF = min(15.01, 13.92) = 13.92 BB
              ⇒ Hero COVERS BTN by 1.09 BB.

--------------------------------------------------------
PREFLIGHT-2: BOUNTY DIRECTION MATRIX (PKO / BOUNTY / MYSTERY_BOUNTY)
--------------------------------------------------------

Required format (table, before any bounty math):

    [Bounty Matrix]
    | Opp | Bounty% | Stack vs Hero | Direction if Hero wins |
    |---|---|---|---|
    | each opponent in pot or yet-to-act | their % | covers/covered | +Hero / -Hero |

Rules:
  • Hero collects opponent's bounty (positive for Hero) iff Hero's
    pre-shove stack >= opponent's pre-shove stack — i.e., Hero
    covers them.
  • Hero loses Hero's own bounty value to whichever caller busts
    Hero — relevant only if a covering opponent calls.
  • A SHOVE has mixed direction when Hero covers some opponents
    and is covered by others. Single-direction shorthand ("Hero
    has the biggest bounty so all callers profit from busting
    Hero") is WRONG in this case and must be replaced with the
    per-opponent matrix.
  • Detection-side hint: the bounty asymmetry that matters for
    EV is the WEIGHTED direction across LIKELY callers. A bounty-
    negative matrix dominated by deep stacks that will fold doesn't
    cost Hero much; a bounty-positive matrix where the most likely
    caller is the short stack Hero covers is a real EV gain.

Worked example (KTo CO 15BB hand, 2026-05-12; Hero bounty 22%):
  [Bounty Matrix]
  | Opp | Bounty% | vs Hero | Direction |
  |---|---|---|---|
  | UTG (in pot) | 15% | covers Hero | -Hero if UTG wins |
  | HJ (in pot)  | 15% | covers Hero | -Hero if HJ wins  |
  | BTN (yet)    | 16% | Hero covers | +Hero if Hero wins |
  | SB (yet)     | 16% | covers Hero | -Hero if SB wins  |
  | BB (yet)     | 16% | covers Hero | -Hero if BB wins  |

  ⇒ Mixed. Bounty-positive vs BTN (the most-likely caller given
    BTN's eff-stack price). Bounty-negative vs everyone else. The
    blanket "Hero has biggest bounty so all callers are paid to
    bust Hero" framing FAILS this matrix — directionally wrong
    vs the most likely caller.

--------------------------------------------------------
WHY THIS EXISTS
--------------------------------------------------------

The v7.47 trigger: KTo CO 15BB squeeze-shove vs UTG-open + HJ-flat
in PKO. Initial analysis used Hero=15.01 as eff stack (BTN=13.92
was shorter and yet-to-act; correct eff was 13.92, Hero covered
BTN by 1.09). Initial analysis also framed bounty unidirectionally
("Hero has biggest bounty, all callers profit from bust") — wrong
for the BTN matchup specifically.

Verdict didn't change (KTo is still a fold here — dominated multi-
way), but the reasoning was incomplete and the bounty piece was
directionally wrong in part. Process discipline > smart reasoning:
the forcing function makes the miss impossible, smart reasoning
allowed it.

MANUAL ADD PROTOCOL (v7.21 — DON'T promote losing hands to punts)
========================================================

When the analyzer says "0 punts" or "0 mistakes" and Claude is tempted
to manually flag a losing hand as a punt/mistake, Claude MUST FIRST:

1. RECOGNIZE THE FAILURE MODE: Big losing hands (PF_PFR_allin_lost,
   3BP_PFR_allin_lost) are usually one of three things:
   - COOLER: standard play, ran into top of villain's range
   - VARIANCE: +EV decision, bad runout
   - READ-DEPENDENT DECISION: defensible vs Hero's actual read,
     even if -EV vs default/tight range

   Default classification for losing PFR all-in hands = COOLER or
   READ-DEPENDENT, NOT punt. Claude must overcome this default to
   manually-add as a punt.

2. RUN THE FULL ANALYTICAL CHECKLIST (mandatory for ANY preflop
   near-all-in spot before classifying):

   A. POT ODDS — Calculate chip equity needed:
      - amount-to-call / total-pot-including-call
      - Show the arithmetic explicitly, not a guess

   B. VILLAIN RANGE — With context tag:
      - standard / squeeze / cover_jam / micro / read-driven / wide-pop
      - If context widens (e.g., chip-leader cover, squeeze-jam, micro
        all-in), USE the wider range. Don't apply base chart blindly.
      - When a player-specific read is stated, USE Hero's read as
        primary range, not population default.

   C. BOUNTY ADJUSTMENT — Mandatory for every bounty hand:
      - Does Hero cover villain? → Hero gains bounty premium when
        winning. Reduces required equity by 3-7pp typically.
      - Does villain cover Hero? → Hero loses bounty when busting.
        Required equity stays at chip-EV level (no bounty discount
        for Hero, but villain's call range is wider so Hero's FE
        is lower).
      - $32 bounty King ≈ $5-10 bounty value ≈ ~10BB at start.
        Premium scales with stack-to-bounty ratio.

   D. EQUITY vs RANGE — Combo-weighted equity:
      - Enumerate combos for each villain hand (account for Hero's
        blockers reducing combo counts: Hero with As blocks all
        AKo/AQo/AJo combos containing As, etc.)
      - Compute Hero equity vs each combo class
      - Weighted average = (Σ combos × eq) / total combos
      - Show the breakdown in a table, not just the final number

   E. BLOCKERS — Critical for jam-with-blockers spots:
      - What does Hero block from villain's value range?
      - KQs blocks KK (1 combo of 6), AK (4 combos of 16), AQ (4
        combos of 16) → significant unblock of villain's calling/
        value range
      - AQo blocks AA (3 combos of 6), AK (8 combos of 16), AQ
        (10 combos of 12) → reduces dominators in villain range
      - Blocker effects shift the EV by 1-3pp typically; meaningful
        in close decisions

   F. FOLD EQUITY — Only for aggressor (jammer/raiser):
      - 0% FE when Hero is calling all-in (skip this step)
      - When Hero is jamming: estimate villain's fold % to the jam
        based on villain's range commitment (e.g., villain who 3-bet
        to 6.4BB at 41BB stack folds 50-60% to a 4-bet jam if their
        range had bluffs; folds 0-20% if pure value)
      - EV(jam with FE) = FE × pot_pickup + (1-FE) × showdown_EV

   G. EV CALCULATION — Always vs the alternative:
      - EV(action) − EV(best alternative) = decision EV
      - Show in chips AND BB
      - Show across PLAUSIBLE villain range bands:
        "vs tight (TT+/AK): -10BB EV
         vs Hero's stated read (88+/AQo+/AJs/ATs+): -3BB EV
         vs wider (66+/AJo+): +1BB EV"
      - This lets Ron decide based on his actual read

   H. ICM CONTEXT — Mandatory when tournament_phase ∈
      {bubble_zone, post_bubble, ft_zone}. Skip for late_reg
      and post_reg. (Added v7.25 — incorporates J41/J42/J43.)

      H1. DIRECTIONALITY (J41) — Identify aggressor and responder:
          - Big stack OPENS / 3-bets → big expands ranges, short
            should NARROW response (>50% in some bubble scenarios)
          - Short stack JAMS → short stays tight; big stack
            RESPONDER ALSO NARROWS (counterintuitive but correct;
            big has no fold-equity-as-aggressor benefit when villain
            is already all-in, and ICM penalizes losing flips even
            with survival)
          - Verify chip_leader_cover_jam tag matches the actual
            direction. The tag widens Hero's call by 15-20pp ONLY
            when Hero is the SHORT STACK calling a covering jam.
            Do NOT invert it for big-stack-as-responder.

      H2. LADDER VALUE (J42) — Estimate ladder $ value:
          - Each pay jump ≈ 2x buy-in worth
          - Count pay jumps within reach (1-3 typical at FT)
          - Cumulative ladder $ may exceed Hero's chip-EV +EV
          - Rule of thumb: a chip-EV +EV call near a pay jump can
            be -EV in $ if it busts ≥40% of the time and 1+ jumps
            are within reach
          - Conversely: a "punt-looking" fold can be CORRECT when
            ladder protection > marginal chip-EV gain

      H3. RAISE-OR-FOLD LENS (J43) — For any flat-call in ICM:
          - Was the hand strong enough to call profitably with
            no fold equity claimed?
          - Was 3-bet (raise) or fold the better line?
          - Exceptions: BTN closing action, BB getting good odds,
            big stack vs big stack (chip-EV dominates)
          - When flat from CO/HJ/MP in bubble+, default lens is
            J43 — flag for review even if EV math is borderline

      ICM verdict: If H1 directionality is wrong OR H2 ladder
      value contradicts chip-EV OR H3 says flat was wrong line,
      lean toward MISTAKE classification (not MARGINAL).

3. ASK BEFORE PROMOTING: If Claude wants to manually-add a hand to
   Confirmed Mistakes that the analyzer didn't flag, Claude MUST
   FIRST ASK Ron about:
   - Villain reads (was villain known to be wide/tight/specific
     tendency?)
   - Whether bounty cover applied (and which direction)
   - Whether it was a population read or a player-specific read
   - Whether stacks-behind / table dynamics changed the decision
   - For ICM phases: H1/H2/H3 verdicts

4. ONLY MANUALLY-ADD AS PUNT IF:
   (a) The hand violates a hard rule (e.g., 4-bet jamming AAo at
       100BB, limp-folding SB BvB)
   (b) EV is dominantly negative across ALL plausible villain
       ranges (tight → wide), even with bounty + blocker adjustments
   (c) Ron has explicitly confirmed it was a misplay

5. NEVER CLAIM "DETECTOR GAP" without first verifying the decision
   is actually -EV across reasonable range assumptions. The detector
   correctly NOT flagging a defensible read-dependent decision is the
   correct behavior, not a bug.

6. NEW DETECTOR PROPOSALS — High bar (v7.21 detector taxonomy):
   The cross-product of action types × severity × context = dozens
   of potential detectors. DO NOT propose narrow per-spot detectors
   when the pattern is read-dependent.

   Build a new detector ONLY when:
   (a) The pattern is detectable from hand structure alone (no
       villain-read input required)
   (b) The pattern is dominantly -EV across ALL plausible ranges
       (i.e., it's a hard-rule violation)
   (c) The pattern occurs in 3+ sessions

   Defensible new detectors: "Hero opens AAo/KKo at 100BB, faces
   3-bet, folds" (always wrong); "Hero limp-folds at SB BvB
   short-handed" (always wrong).

   NON-detectors (handle via review framework, not new detector):
   "Hero 4-bet jams light at sub-30BB" (depends on villain range,
   blockers, FE); "Hero calls cold-jam wide" (depends on villain
   range, bounty, coverage); "Hero squeezes wide MW" (depends on
   opener tendencies + bounty incentives).

   Rule: when in doubt about a new detector, route through the review
   framework + exception log instead. The exception log is the
   correct memory layer for read-dependent patterns.

This protocol exists because Claude has historically over-classified
read-dependent decisions as punts, generating false-positive "session
story" narratives that aren't supported by the actual EV math.

SAMPLE-SCOPING GUARDRAIL (v7.22 — applies to ALL rate metrics):

Every rate-based stat has an IMPLICIT depth/context where its target
range was derived. The denominator MUST be scoped to that context, or
the metric is garbage. The F2-3B fix (v7.22) was one instance of a
broader principle:

A target range like "F2-3B 55-65%" is derived from population behavior
at 25-100BB where a 3-bet is a conventional raise. It does NOT apply
to:

CONFIDENCE-AWARE STAT SIGNALS (v7.22 — Wilson 90% CI):

STATISTICAL METHOD:
  - Test: Wilson score interval for binomial proportions
    (strictly better than Wald for small n and values near 0/100)
  - Alpha: 90% (z = 1.645) — pragmatic default for coaching decisions:
      95% CI produces too many "can't tell" verdicts at our sample sizes
      80% CI is too permissive — labels noise as confident
      90% flags real issues without drowning in noise
  - Minimum n: 10 for any CI classification. Below that → ⚪ (no signal).
  - Four states are ONLY distinguished by: is n ≥ min? Then where does
    the CI sit relative to the target?

FOUR-STATE SEMANTICS:
  🟢 green   — CI fully inside target range (confident in target)
  🔴 red     — CI fully outside target range (confident out of target)
  🟡 yellow  — CI overlaps exactly one target boundary
               (genuinely borderline: have data, but on the edge)
  ⚪ neutral — insufficient sample, no claim made

TEST TYPE DEPENDS ON TARGET SHAPE:

  Range (e.g., WWSF 42-48%): TWO-SIDED
    CI fully in [low, high]                → 🟢
    CI fully below low OR fully above high → 🔴 (both are "out of range")
    CI overlaps a boundary                 → 🟡

  Threshold ≥ target (e.g., SD Aggressor ≥ 40%): ONE-SIDED MIN
    CI lower bound ≥ target  → 🟢
    CI upper bound < target  → 🔴
    Else                     → 🟡

  Threshold ≤ ceiling (e.g., VPIP-PFR Gap ≤ 4%): ONE-SIDED MAX
    Invokes _clr with invert=True.
    CI upper bound ≤ ceiling → 🟢
    CI lower bound > ceiling → 🔴
    Else                     → 🟡

WHY THIS MATTERS:

A stat at small sample with ambiguous CI is not penalized — it shows ⚪
and doesn't count toward tier promotion/demotion. A stat at small
sample with extreme value (CI fully outside range) IS counted — because
the observation is meaningful even at low sample.

"0 red signals" out of 3 reliable stats is NOT the same as "0 red" out
of 8 reliable stats. The tier classifier uses:
  green_rate = n_green / (n_green + n_red)  — excludes yellow & neutral
  red_rate   = n_red   / (n_green + n_red)

Tracked stats (each with own min sample threshold):
  - SD Aggressor pct       (sd_total ≥ 5)
  - WWSF                   (wwsf_total ≥ 5)
  - Non-SD Win             (wwsf_total ≥ 5)
  - VPIP-PFR Gap non-blind (N ≥ 100, strong at N ≥ 200)
  - Caller IP Flop Agg     (caller_n ≥ 5)
  - F2-3B                  (ftb_opps ≥ 5)
  - Aggression Factor      (N ≥ 150 neutral, ≥ 300 confident)
  - ATS                    (ats_opps ≥ 10)

Verdict block displays:
  "Stat health (confidence-adjusted): X/Y green / X/Y red /
   Z stats neutral (insufficient sample) — red: stat_names"
so the classification is auditable.

ISO-RAISE vs ISO-JAM (v7.20 — DISTINCT CONCEPTS):

These were previously conflated in the analyzer. They are different
decisions with different thresholds.

  ISO-JAM: Hero goes all-in (≥95% of starting stack committed) raising over
           a villain jam. No fold equity retained; pure showdown decision.
           Thresholds are TIGHT — defined by equity vs villain's call-the-
           iso-jam range.
    vs EP jam: TT+, AQs+, AKo
    vs MP jam: 99+, AJs+, AQo+
    vs LP jam: 77+, ATs+, AJo+, KQs

  ISO-RAISE: Hero raises over a villain jam to isolate, WITHOUT going all-
             in. Typical size 2-3x the jam amount. Keeps stack behind for
             fold equity vs players behind + flexibility to fold to a 4-bet
             jam. Thresholds are 2-3x WIDER than iso-jam thresholds because
             Hero retains fold equity.
    vs micro-jam (<5BB): any reasonable hand (A-high, K-high, pair, suited
                          broadway, suited connector, any bounty cover spot).
    vs short-jam (5-12BB): widened from iso-jam by ~10 combos per jammer pos.

  The analyzer determines iso-jam vs iso-raise via `pf_allin` field
  (canonical) or hero_committed_bb / stack_bb ≥ 0.95.

CHIP-LEADER COVER-JAM CONTEXT (v7.20 detector + J41 principle):

At short-handed final-table (≤5 players), when chip leader (eff > 3x
average) jams into short stacks, their range is significantly wider than
GTO charts assume:
  - Standard GTO BTN open range at 40BB: ~25-30%
  - Chip-leader cover-jam at FT Hyper, short-handed: ~35-50%
  Reasons: bounty collection incentive, stack-preservation less important
  for chip leader, short blinds are forced-defenders, hyper escalation.

  Corollary: defend-vs-cover-jam ranges MUST widen by 15-20pp vs the QR
  base chart. K9o SB vs 43BB BTN cover-jam at 5-handed FT is a CALL, not
  a fold, despite failing the "vs LP jam ≤15BB" chart.

<8BB PUSH/FOLD (heuristic, no charts):
  SB/BB <5BB: push any Ace, King, Queen, pair, suited, connected
  SB/BB 5-8BB: push Ace, King, pairs, suited Q+/connected
  BTN <5BB: push ~70% (Ace, King, suited, broadway, J+x)
  CO <5BB: push Ace, pairs, suited King, broadway

KEY JAKA K-RULES (Apr 12-16, "Mastering Advanced Flop Play"):
  K1: IP raises c-bets → value=OP+ & TP best kickers, bluffs=FDs>SDs
     At <30BB: STOP raising draws that can't call a jam
     Multiway: NO raising (or max 5% if board misses BB)
     vs over-cbetter: WIDEN CALL range, don't raise more
  K2: "Make them FEAR your checks" — OOP PFR default = CHECK
     Gentlemen (MP vs MP): check heavy, respect x/r
     Warriors (MP vs LP, LP vs LP): more aggression, size UP
  K3: IP caller stab when checked to = 40-60% STABLE across textures
     Default: small & wide. 2 high-card boards: big & selective
     "Float → Turn Attack" when OOP checks turn = very high freq
  K4: SRP vs SB c-bet% — earlier position = more betting, shallower = more
     BTN@60BB: 48% bet  | BTN@30BB: 53% bet
     CO@60BB:  56% bet  | CO@30BB:  61% bet
     MP@30BB:  70% bet
     Paired = attack. Monotone + connected = caution.
  K5: Range adv → WIDE+SMALL. Nutted adv only → NARROW+BIG. Monotone → NARROW+SMALL
     OOP PFR must x/r ~10% or checking range is exploitable
  K6: Flop leading is RARE (~2.5% baseline). Specific spots only:
     BB vs IP low straights (765, 643) any depth
     BB vs IP low paired (665, 77A) 60BB+
     3BP low straights 60BB+ / mid-connected (JT7) 30BB-
     Merged lead = SMALL (20-33%). Polar lead = BIG (67-100%)

KEY MECHANICSOFPOKER M-RULES (Apr 20, 25-rule podcast extraction):
  M1: Probe/delayed-cbet exploit — population turn CR rate <2% vs
      delayed cbet. After flop X/X as PFR, MINBET TURN RANGE on
      non-equity-shift turns. DIRECT FIX for Ron leak #1
      (check-call-call-showdown OOP PFR). AUTO-DETECTED.
  M2: Can't hop off the exploit train — if preflop was exploitative,
      all postflop streets must stay exploitative. Ranges shift.
  M3: Plan every street before betting — answer (a) which turns
      continue, (b) which give up, (c) river plan on continuers,
      BEFORE the flop bet. No plan = don't bet.
  M4: Float requires barrel plan — every float needs an explicit
      turn-bluff plan. Addresses Ron leak #2 (Caller IP aggression).
  M5: Mixed turn bluff = committed river — if entering a mixed
      turn bluff, must triple-barrel the right rivers. Don't mix
      if tired/tilted.
  M6: 3BP double-FD turn: barrel AIR, CHECK equity hands.
      Counter-intuitive but correct — equity must fold to the CR
      spike (>8% on these turns) and lose equity. Air loses
      nothing. AUTO-DETECTED. Addresses Ron leak #3 (one-and-done).
  M7: Turn barrel card selection — pick cards that add equity to
      Hero's range without adding NUTS to villain's range.
      Example: 543 flop → 7 turn = good (adds gutters, no nuts);
      8 turn = bad (adds 87/76 two-pair nuts for villain).
  M8: ICM short stack on equity-shift textures (K56fd, A45fd type)
      = REDUCE c-bet freq, check back even nut-level hands.
      Refines J14 with ICM caveat (J14 unchanged for chip-EV).
  M9: CR frequency is elastic to villain's 3bet-over-CR rate.
      Vs non-3-betters, check-raise WIDER than Jaka default.
  M10: Big stack ICM = splashier — bet wider/bigger vs
      risk-premium villains. Bounty amplifies.
  M11: Population river RAISES (not shoves) are value-heavy by
      ~60% vs GTO. Fold bluff-catchers below 2P without blocker.
  M12: Blockers matter proportional to villain's value-combo
      count. Wide range → blockers matter less. Refines J27.
  M13: Bluff-catch best/worst-case framework — fold if best
      case ≈ breakeven and worst case = stack loss.
  M14: Solver EV diffs <2% of pot = noise. Use exploit context
      to break indifferent decisions.
  M15: Study priority: RIVER > TURN > FLOP (mistakes on flop
      absorbed by later-street solver play; rivers are pure loss).
  M16: Multiway vs calling station: SKIP BLUFFS entirely, value
      bet wider+bigger. Balance is non-concept vs station.
  M17: BB defense vs aggressive opener = tighten ~10-15%.
      Drop offsuit connectors first. Extends J13.
  M18: Flat 3-bets WIDER vs MTT 4bet-or-fold meta with T9s/98s/
      J9s. Enters 3BPs where villain is capped at JJ-down.
      50-100BB only (Amit rule dominates below).
  M19: Runout-type bluff bias: LOW-TO-HIGH/flush-complete/3-
      broadway rivers = OVERBLUFFED. Brick rivers + paired
      boards = UNDERBLUFFED. Parser tags runout type.
  M20: Middle sizing (50-66%) = merged middle-strength from
      population. Widen call range vs middle sizings.
  M21: Villain raise-showdown% = raise-call% → villain folds

  M22: Avoid tougher regs when possible (table selection).
