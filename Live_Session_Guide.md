# Live Session → GG Hand History Conversion

**When this applies:** Ron has live tournament notes (typed or pasted from
WhatsApp / phone notes / etc.) and wants them converted to GG-format hand
history files for **GTO Wizard** upload. This is **not** the standard GEM
pipeline — GEM parses online GG hand histories that already exist as `.txt`
files. This workflow goes the other direction: prose notes → GG `.txt`.

**This is a separate workflow.** Don't confuse with `gem_parser.py` /
`gem_analyzer.py` / etc. Those operate on already-formatted hand histories.
Live conversion uses **`live_to_gg.py`** + a session-specific specs file.

---

## Files

| File | Role |
|---|---|
| `live_to_gg.py` | The converter library. Reusable, never edit per session. |
| `live_session_template.py` | Starter template for a new session's specs. Copy + rename per session. |
| `live_session_<DATE>.py` | Session-specific specs file (per session, generated by Claude). |
| `Live_Session_Guide.md` | This file. |

## Workflow

1. Ron pastes/uploads his session notes.
2. Claude copies `live_session_template.py` → `live_session_<YYYYMMDD>.py`.
3. Claude reads the notes hand-by-hand and writes one spec dict per hand.
4. Run `python3 live_session_<YYYYMMDD>.py` — outputs GG-format `.txt`.
5. Ship to `/mnt/user-data/outputs/` and `present_files`.
6. Ron uploads to GTO Wizard.

## Action DSL (used inside each spec's `actions` list)

Position-keyed actions. All position abbrevs: `UTG, UTG1, UTG2, MP, LJ, HJ,
CO, BU, SB, BB`. Note: 7-handed has no LJ; 8-handed has no UTG2; 6-handed has
no UTG1/MP/LJ.

```
"<POS>: fold"           → folds
"<POS>: check"          → checks
"<POS>: call"           → calls (amount auto-computed from cur_bet)
"<POS>: limp"           → SB completing preflop (calls BB amount)
"<POS>: bet 1500"       → first agg on a street
"<POS>: raise 4500"     → raises TO 4,500 total (delta auto-computed)
"<POS>: shove"          → all-in for remaining stack
"<POS>: show Kc Kd two-pair-Kings-and-Tens"  → showdown reveal + hand desc
"F Kh Js 9c"            → deal flop
"T Td"                  → deal turn
"R 8c"                  → deal river
"FOLDS_TO_HERO <POS>"   → fold all positions before Hero (pre-flop only)
"WIN <POS>"             → mark winner (only needed at multi-way showdown)
```

## Spec dict structure

```python
{
    'hand_id': 'LV2604010001',          # synthetic; LV<MMDD><day><nn>
    'tourney_id': '900000001',          # synthetic; one per tournament
    'buyin': '$1 Live National Cup',    # buy-in label
    'level': 5,
    'sb': 200, 'bb': 400, 'ante': 0,
    'dt': '2026/04/22 12:30:00',        # YYYY/MM/DD HH:MM:SS
    'table': 'NC2',
    'n': 9,                             # table size (2-9)
    'btn': 'BU',                        # button position label
    'day_id': '01a',                    # 3-char prefix for player IDs
    'hero': ('SB', 41000, 'Kc 9d'),     # (pos, chips, hole)
    'villain_default_chips': 30000,     # starting stack for unspecified villains
    'villains': {'BB': 42000},          # override stacks for specific positions
    'actions': [                        # DSL list, in order
        'FOLDS_TO_HERO SB',
        'SB: limp', 'BB: raise 1500', 'SB: call',
        'F Kh Js 9c',
        'SB: check', 'BB: bet 400', 'SB: raise 2000', 'BB: call',
        'T Td',
        'SB: check', 'BB: bet 8000', 'SB: fold',
    ],
}
```

## Critical lessons (from past sessions)

### 1. "Folds to me in SB" is NOT heads-up

In live multi-handed games, "folds to me in the SB" means everyone before SB
folded — but it's still a 9-max table where postflop SB acts FIRST OOP, not
HU button-IP. Always set `'n': 9` (or whatever the actual table size is) and
use `FOLDS_TO_HERO SB` to fold positions UTG–BU, then SB acts.

**Wrong:** `'n': 2, 'btn': 'SB'` — gives Hero IP postflop. ❌
**Right:** `'n': 9, 'btn': 'BU', FOLDS_TO_HERO SB` then SB OOP. ✓

### 2. Position labels match table size

7-handed layout: `[UTG, MP, HJ, CO, BU, SB, BB]` — no `LJ`, no `UTG1`. If
Ron's note says "HJ opens", that's the 3rd seat in 7-max. Match the layout.

### 3. Verify, don't fabricate

If Ron's note is ambiguous (cards not stated, stacks unclear, action gaps),
flag it and ASK or use a clearly-marked synthetic value. Don't silently
invent details.

### 4. Filter scope for GTO Wizard

By default, **skip pure preflop fold-outs and jam-folds** (e.g. "Hero folds
AKo to UTG open" or "Hero shoves 8BB QQ, called by KK"). They have zero
postflop content for GTO Wizard. **Always ask Ron** whether he wants
completeness (all hands) or analyzability (postflop only).

### 5. Known converter limitations

- **Chops/split pots:** Not supported. Mark one player as winner and add
  inline comment.
- **Side pots:** Not modeled (multiple all-ins for different amounts will
  produce a single pot).
- **Ante=0 hands:** Render as `Level1(100/100(0))`. If GTO Wizard rejects,
  switch to a token ante (e.g. 50).
- **Player IDs** are 8-char synthetic strings (e.g. `02abb_`). Format-strict
  parsers may complain.

## Naming conventions for output files

`Claude_GEM_2_GTOWizard_<YYYYMMDD>_V<N>.txt` — same convention as the
standard GEM GTO Wizard export, even though this is a different pipeline.
This keeps Ron's GTO Wizard history organized.

## Speed expectations

- First session using this workflow: ~45–60 min for ~40 hands (all spec
  writing).
- Subsequent sessions: ~20–30 min — workflow is muscle memory, just spec
  writing.
- Per-hand: ~30 sec (simple) to 2 min (multiway w/ showdown).

If a session takes longer than this, something is wrong — flag it.
