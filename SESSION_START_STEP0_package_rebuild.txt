SESSION-START STEP 0 — EXTRACT THE GEM RUNTIME BUNDLE   (v9-capacity, CHAT)
================================================================================
The project carries the ENTIRE runtime as ONE self-extracting file:

    gem_src_bundle.py          (prints its BUNDLE_VERSION on extract)

Source files are NOT stored flat in the project anymore. DO THIS FIRST,
before reading other files or running any pipeline step:

1. Extract the runtime, then pull in the flat session-data CSVs and the
   equity engine:
       python /mnt/project/gem_src_bundle.py /home/claude/gem
       cd /home/claude/gem
       cp /mnt/project/session_*.csv . 2>/dev/null || true
       pip install --quiet phevaluator 2>/dev/null || \
           pip install --quiet phevaluator --break-system-packages || true
   Expect: "GEM bundle <version>: extracted ~87 files" and a complete tree
   INCLUDING the gem_report_draft/ package directory. The old 16-file
   manual package rebuild is OBSOLETE — the bundle preserves structure.
   The pip install is REQUIRED for exact EAI equity / True EV: without
   phevaluator the analyzer degrades to a rank heuristic (True EV moves
   several BB/100), WARNS loudly on stderr, and the report marks True EV
   "approximate" — do not ship that without telling Ron.
   The CSV copy matters: session_financials*.csv and session_history*.csv
   live FLAT in the project (Ron refreshes them between releases) and the
   resolvers glob the working directory only. If no session CSVs exist,
   trend/skill sections render standalone — say so, don't improvise.

2. Verify before proceeding:
       python -c "import gem_report_draft, gem_analyzer; print('imports OK')"
       python verify_release.py --project-dir .
   Both must pass: expect a clean v8.15.0 copy — 42/42 files, 376 canaries,
   10 anti-canaries (the count grows as modules/canaries are added each release).
   On any failure STOP and tell Ron — do not improvise a report.

3. THE CANONICAL PIPELINE ENTRY POINT IS:
       python gem_analyzer.py <SESSION_DIR>
   That is the ONLY production runner. It parses, analyzes, builds the
   coverage layer (pot odds, coaching cards, candidates + the analyst
   WORKSHEET, auto-verdicts, coverage gate) and writes versioned outputs
   (Pokerbot_<Player>_<dates>_V<N>.html). After writing the analyst file
   (session_analysis_<dates>.json, OUTSIDE the HH input dir), re-render
   with `python gem_analyzer.py <SESSION_DIR> --quick` (cache-guarded;
   it hard-aborts if the cache is from a different session).
   Do NOT use any other runner or hand-rolled render driver — if a
   helper script looks like a pipeline runner, it is not; gem_analyzer
   is the pipeline.

3b. Work from /home/claude/gem for ALL pipeline steps. Session uploads
   (hand histories etc.) extract to sibling dirs as usual.

   GAME SUMMARIES ARE PART OF THE RUN: Ron's session uploads normally
   contain TWO zips — (A) hand histories and (B) tournament game
   summaries. Zip B carries cash / net / ROI / finishes and resolves
   flighted advancements. Extract it to a sibling directory — discovery
   is automatic (content-sniffed, v8.12.5). If only hand histories were
   uploaded, say so explicitly in the session preamble ("no game
   summaries attached — cash/ROI columns will be blank").

4. Prose references stay FLAT in the project — read them directly from
   /mnt/project/: GEM_Quick_Reference.txt, Analyst_Writing_Checklist.md,
   the reference docs, GEM_Changelog.txt. (QR/changelog/this file also
   ride INSIDE the bundle so the verifier can hash them; the flat copies
   are the ones you read.)

5. Analyst-file rule unchanged: --analyst-file must NOT point inside the
   hand-history input directory.

RELEASE NOTE: a release is now ONE artifact — a new gem_src_bundle.py
(plus flat prose re-uploads only when prose changed). If verify_release
reports stale files after an update, bundle and prose came from different
drops — re-upload both from the same release.
