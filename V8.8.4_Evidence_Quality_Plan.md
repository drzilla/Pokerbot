## Architecture context (for GPT review)

### How evidence flows from data to UI

```
Parser → hands[]
  → gem_villain_intel.py:
      build_evidence_atoms() → atoms_by_villain{vk: [atom, ...]}
        Each atom has: dimension, signal, badge, evidence_text, read_impact, hero_involved
      _build_read_states() → read_states{vk: {primary_read, dimension_scores, ...}}
        Uses ALL atoms + _DIMENSION_MAP to score dimensions → pick primary_read
      detect_exploit_opportunities() → exploit_opportunities[{...stamped fields...}]
        Each detector calls _villain_has_read() (prior atoms + temporal gate)
        Then _stamp_exploit_read(exp, detector_name, source)
  → gem_report_draft/sections_iv_xii.py:
      Matrix table: groups exploits by exploit_read_label (v8.8.3 fix)
      Evidence links: call openReadEvidence(readLabel)
  → gem_report_draft/sections_xiv.py:
      Serializes to window.villainIntel and window.exploitOpportunities
  → gem_report_draft/_html.py:
      openReadEvidence(readLabel) → filters villainIntel by archetype_label
      openExploitDrilldown(readLabel, filter) → filters exploitOpportunities by read_label
```

### How atoms map to reads via `_DIMENSION_MAP`

```python
_DIMENSION_MAP = {
    'loose_passive': {'loose': 2, 'passive': 2},
    'sticky':        {'sticky': 4},
    'tight':         {'tight': 3},
    'pivot':         {'aggressive': 3},    # pivots score into aggressive
    'aggressive':    {'aggressive': 4},
}
```

Each atom has a `dimension` field. When building read states, each atom's dimension is mapped through `_DIMENSION_MAP` to get scored read dimensions. The primary_read is the highest-scoring dimension that exceeds its threshold.

**This is the key insight for G1:** An atom with `dimension='pivot'` maps to `aggressive` via `_DIMENSION_MAP`. So `pivot` atoms ARE supporting evidence for Aggressive reads. But atoms with `dimension='loose_passive'` (limps, limp-calls) do NOT map to aggressive — they support Loose Passive. Currently `openReadEvidence('Aggressive')` shows ALL atoms from villains classified as Aggressive, including their limp atoms. GPT correctly flagged this as confusing.

### Dimension-to-read support mapping (derived from `_DIMENSION_MAP`)

| Atom dimension | Supports read | Signals |
|---|---|---|
| `aggressive` | Aggressive | 3-bet, bluff, pressure raise |
| `pivot` | Aggressive | Passive-to-aggro pivot (strong aggro signal) |
| `tight` | Nit / Rock | Overfold, tight fold patterns |
| `loose_passive` | Loose Passive | Open limp, limp-call, donk, minbet, cold-call |
| `sticky` | Sticky Passive | Showdown with weak hand, sticky calls |

### Current evidence drilldown filters

The `buildReadEvidenceTable` function in `_html.py:2018` supports these filters:
- `all` — show all atoms
- `hero involved` — only atoms where Hero was in the hand
- `pivots` — only pivot-badged atoms
- `notes` — only note-badged atoms
- `reviewable only` — only atoms from reviewable appendix hands

Missing: **no filter by dimension/read-support**.

### Fix F1: Evidence drilldown — partition by supporting dimension (GPT G1)

**Problem:** `openReadEvidence('Aggressive')` in `_html.py:1913` filters by `v.archetype_label === readLabel`, then shows ALL atoms from matching villains. An Aggressive-classified villain with 24 open-limp atoms shows those limps under "Aggressive evidence" — confusing.

**Root cause:** Filtering is at the villain level, not the atom level.

**Proposed fix:** Partition atoms into two groups before rendering:

1. **Supporting evidence** — atoms whose `dimension` maps to this read via `_DIMENSION_MAP`:
   - Aggressive: `dimension in ['aggressive', 'pivot']`
   - Nit / Rock: `dimension in ['tight']`
   - Loose Passive: `dimension in ['loose_passive']`
   - Sticky Passive: `dimension in ['sticky']`

2. **Other observed signals** — all remaining atoms from the same villains.

**Implementation (JS in `_html.py`):**

```
a) Add a constant mapping in the JS preamble:
   var READ_SUPPORT_DIMS = {
     'Aggressive': ['aggressive', 'pivot'],
     'Nit / Rock': ['tight'],
     'Loose Passive': ['loose_passive'],
     'Sticky Passive': ['sticky']
   };

b) In openReadEvidence(), after collecting allAtoms:
   var supportDims = READ_SUPPORT_DIMS[readLabel] || [];
   var supporting = allAtoms.filter(a => supportDims.indexOf(a.dimension) >= 0);
   var other = allAtoms.filter(a => supportDims.indexOf(a.dimension) < 0);

c) Add a new filter option 'supporting' to buildReadEvidenceTable.
   Default view: supporting only.
   Add filter button: "Other signals from these villains (N)"
   When clicked, shows the full unfiltered view (current behavior).

d) Update the header counts to show:
   "N supporting signals | M other signals from same villains"
```

**Acceptance test (from GPT):**
- Click Nit/Rock Evidence → default rows show Repeated Blind Overfold / tightness. Not open limps.
- Click Aggressive Evidence → default rows show pivots / raises / aggression. Not open limps.
- "Other signals" filter/toggle reveals the full atom set.

**Risk:** Low. Additive partition, no data change. Existing "all" filter still shows everything.
**Effort:** ~1 hour. JS-only change in `_html.py`.

**Decision needed:** Is the partition approach sufficient, or does GPT want a full "Signal / Why it matters / Adjustment" coaching format per atom (G6)? If coaching format is needed, it's a separate item requiring evidence_text + read_impact enrichment in the Python atom builder — significantly more work.

### Fix F3: Appendix prioritization for exploit hands (GPT G4)

**Problem:** 11 missed exploits, only 5 are reviewable (appendix-included). Missed exploits are high-value coaching moments but get no inclusion priority.

**Root cause:** The appendix builder in `gem_report_data.py` selects hands by a priority system (punts > mistakes > coolers > ...). Exploit hands have no special priority — they compete with all other hands for limited appendix slots.

**Proposed fix:**

```
In gem_report_data.py appendix builder:
  1. Add exploit_opportunity hands to the priority list, after punts/mistakes
     but before generic notable hands.
  2. Weight: missed_exploit severity B (same as current detector severity).
  3. Cap: at most 15 exploit hands in appendix (prevents dominating).
```

**Alternative (lower effort):** If appendix inclusion is too disruptive, enrich the evidence-only detail panel in the exploit drilldown. Currently non-reviewable exploit rows show minimal info. Add:
```
hand_id, cards (if known), position, street, hero_action,
villain_read_before_decision, recommended_adjustment, evidence_text
```

This way even without full hand detail, the exploit coaching moment is useful.

**Decision needed:** Full appendix priority (changes `gem_report_data.py`) or enriched evidence-only panel (changes `_html.py` JS only)?

**Risk:** Low-Medium. Appendix priority could displace other hands.
**Effort:** 1-2 hours depending on approach.

### Enhancement E1: Coaching-oriented evidence descriptions (GPT G6)

**Problem:** Evidence rows show `Signal: Repeated Blind Overfold` but don't explain why it matters or what adjustment it implies.

**Desired format (per GPT):**
```
Signal: Repeated Blind Overfold
Why it matters: Villain is likely overfolding blinds.
Adjustment: Open wider first-in when this player is in blinds.
```

**Assessment:** This requires adding coaching strings to each atom signal type. Currently atoms have:
- `signal` (machine key, e.g., `overfold_blind`)
- `label` (human label, e.g., `Repeated Blind Overfold`)
- `evidence_text` (sentence, e.g., `Villain folded BB 4 of last 5 opportunities`)
- `read_impact` (dimension impact, e.g., `Tight +3`)

The coaching chain (why + adjustment) would need to be generated per signal type. Options:
1. **Static lookup table:** 15-20 signal types → coaching strings. Add to `_html.py` JS.
2. **Python-side:** Generate at atom creation time in `gem_villain_intel.py`.
3. **LLM-generated (parked):** Per the governance rules, LLM analyst handoff is parked.

**Recommendation:** Static JS lookup table is lowest-risk and sufficient for MVP. Can be iterated.

**Risk:** Low. Additive display change.
**Effort:** 1-2 hours (authoring the coaching text is the main work, not the code).
**Decision needed:** Is this in scope for v8.8.4, or deferred?
